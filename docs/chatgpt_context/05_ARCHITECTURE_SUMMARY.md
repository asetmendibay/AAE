# Architecture Summary

**Назначение:** Краткая выжимка целевой архитектуры для восстановления контекста ChatGPT. Без деталей реализации.

## 1. Основные модули и слои системы

Проект строится по принципам чистой архитектуры в формате модульного монолита. Выделены следующие слои:

- **Core (Ядро):** Определение бизнес-сущностей, портов (интерфейсов) и контрактов. Содержит модель задачи (Job) и её жизненный цикл. Не имеет внешних зависимостей.
- **Application (Приложение):** Оркестрация и выполнение логики жизненного цикла задач. Управляет переключением состояний Job.
- **Infrastructure (Инфраструктура):** Реализация портов ядра. Сюда относится адаптер браузера (Browser Adapter на базе Playwright), провайдер конфигурации, логгер, дисковое хранилище профилей.
- **Interfaces (Интерфейсы):** CLI и любые будущие интерфейсы запуска (API/Dashboard) для взаимодействия с внешним миром.
- **Modules (Модули / Сценарии):** Конкретные изолированные сценарии автоматизации сайтов. Каждый модуль взаимодействует с системой только через стабильные интерфейсы ядра, не зная деталей инфраструктуры.
- **Composition Root (Точка сборки):** Управляет связыванием зависимостей (Dependency Injection) при старте приложения.

## 2. Границы ответственности и направления зависимостей

- **Направление зависимостей:** Все внешние слои (`infrastructure`, `interfaces`, `modules`) зависят от `core` и `application`. Внутренние слои ничего не знают о внешних (например, бизнес-логика не зависит от Playwright).
- **Изоляция браузера:** Прямая работа с Playwright разрешена только внутри инфраструктурного слоя `Browser Adapter`. Никакие селекторы, действия или специфичные типы Playwright не должны проникать в слой `Modules` или `Core` напрямую.
- **Изоляция данных:** Каждая задача (Job) выполняется в строго изолированном контексте (сессии) браузера. Обмен mutable состоянием между параллельными задачами запрещён.

## 3. Ключевые архитектурные принципы

- **Порты и адаптеры:** Любая сторонняя библиотека (логирование, парсинг, HTTP-запросы) должна быть скрыта за интерфейсом. Прямой импорт внешних пакетов в бизнес-логику запрещён.
- **Конфигурация вне кода:** Настройки, селекторы (если они динамические) и параметры запуска не могут быть зашиты в коде модулей.
- **Диагностируемость:** Ошибки должны типизироваться и перехватываться с сохранением трассировки и логов. Любое падение задачи должно приводить к безопасному закрытию ресурсов и формированию диагностических артефактов (скриншоты, трассировка).
- **Неявные ожидания (Антипаттерн):** Запрещено использование фиксированных задержек (например, `sleep(5000)`). Допускаются только динамические ожидания состояний элементов или событий сети.
