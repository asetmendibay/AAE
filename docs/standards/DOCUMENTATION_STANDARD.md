# Documentation Standard (Стандарты документации)

## 1. Общие требования

- **Язык:** Документы общего планирования, правил и ценностей пишутся на русском языке. Технические спецификации API, кода и ТЗ могут содержать англоязычные термины.
- **Формат:** Markdown (GitHub Flavored Markdown).
- **Кодировка:** UTF-8.
- **Источник истины (Source of Truth):** Каждый факт хранится в одном каноническом документе; остальные документы ссылаются на него, а не копируют его. В проекте установлена следующая жесткая иерархия источников истины:
  1. Git Repository (код и слияния)
  2. Documentation (архитектура, Master Plan, стандарты в `docs/core/` и `docs/standards/`)
  3. Executive Context (`docs/executive/`)
  4. ChatGPT Context Layer (`docs/chatgpt_context/`)
  5. AI Sessions (история чатов ИИ никогда не признаётся источником истины)

---

## 2. Использование перекрёстных ссылок

- Используются переносимые относительные Markdown-ссылки.
- Ссылка должна работать после clone в другую директорию и на другой ОС.
- Ссылка из документа в `docs/core/` на `docs/executive/` записывается как
  `../executive/00_EXECUTIVE_CONTEXT.md`; из README — как
  `docs/executive/00_EXECUTIVE_CONTEXT.md`.
- Ссылки проверяются при изменении или переносе документа.

---

## 3. Обновление документации

- Изменение baseline, contract или workflow отражается в том же pull request.
- Project state обновляется при изменении текущего roadmap stage.
- Progress обновляется при закрытии этапа или его переходного критерия.
- Decision journal обновляется только при принятом ADR/решении Product Owner.
- Исторические workspace-записи не переписываются для изменения текущего
  процесса; фактические ошибки ссылок в них могут исправляться отдельно.
