# Project Operating System (Система управления проектом)

## 1. Назначение

Automation Engine является долгосрочным инженерным проектом.  
Данный документ описывает единые правила разработки проекта, хранения документов, истории, рабочих материалов и взаимодействия участников.  
Все ИИ (искусственные интеллекты), работающие над проектом, обязаны строго соблюдать положения данного документа.

## 2. Структура проекта

Вся работа и файлы проекта организованы в следующую древовидную структуру:

```
Automation Engine/
├── docs/
├── workspace/
├── archive/
├── src/
├── tests/
├── scripts/
├── profiles/
├── logs/
└── temp/
```

## 3. Назначение папок

- **`docs/`** — Постоянная техническая и организационная документация проекта.
- **`workspace/`** — Рабочая зона для текущих задач и ежедневных материалов.
- **`archive/`** — Архив завершенных задач и материалов прошлых дней/недель.
- **`src/`** — Исходный код разрабатываемого приложения Automation Engine.
- **`tests/`** — Автоматические тесты (Unit, Integration, E2E).
- **`scripts/`** — Вспомогательные скрипты для автоматизации задач разработки, сборки или деплоя.
- **`profiles/`** — Данные профилей браузеров (куки, локальное хранилище, настройки).
- **`logs/`** — Лог-файлы работы движка и сервисов.
- **`temp/`** — Временные данные, кэш и промежуточные файлы.

## 4. Организация workspace

Текущая работа организуется по датам внутри папки `workspace/` по следующей иерархии:

```
workspace/
└── [Год]/
    └── [Месяц]/
        └── Week_[Номер недели]/
            └── [ГГГГ-ММ-ДД]/
                ├── notes.md        # Заметки
                ├── tasks.md        # Задачи дня
                ├── ideas.md        # Идеи
                ├── ai_prompts.md   # Промпты для ИИ
                └── results.md      # Результаты работы
```

## 5. Правила ежедневной работы

Каждый рабочий день имеет собственную отдельную папку формата `workspace/[Год]/[Месяц]/Week_[Номер]/[ГГГГ-ММ-ДД]/`.  
Каждая ежедневная папка должна содержать следующие файлы:

- **`tasks.md`** — Список задач на сегодня.
- **`notes.md`** — Заметки в процессе выполнения задач.
- **`ideas.md`** — Идеи, возникшие в течение дня.
- **`ai_prompts.md`** — Промпты, использованные для работы с ИИ.
- **`results.md`** — Отчет о результатах дня.

## 6. Завершение рабочего дня

Рабочие материалы остаются в `workspace/` до завершения задачи. Repository
Chief Engineer обновляет project records только по правилам Unified Development
Workflow: при изменении стадии, завершении этапа, принятии ADR или merge.

## 7. Роли и workflow

Постоянные роли определены только в `docs/AI_TEAM/00_TEAM.md`; единственный
workflow — в `docs/core/11_WORKFLOW.md`. Этот документ не дублирует их и не
создаёт альтернативных полномочий.

## 8. Правила изменения документации

Изменение документации выполняется в рамках task specification. Изменение
baseline, структуры каталогов, workflow, ролей или constitution проходит через
ADR либо явное решение Product Owner.

## 9. Git

Git delivery policy и branch rules определены в `docs/standards/16_GIT_WORKFLOW.md`
и `docs/standards/GIT_STANDARD.md`.

## 10. Цель

Новый инженер или ассистент должен восстановить актуальный контекст через
executive documents и canonical sources, не опираясь на историю чатов.

## 11. Project Memory

- Репозиторий и merged history — источники фактов.
- Документация хранит решения, контекст и воспроизводимые правила.
- Ни один ИИ не считается единственным носителем знания проекта.
