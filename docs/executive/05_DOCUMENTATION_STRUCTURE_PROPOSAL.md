# Documentation Structure Proposal

**Статус:** выполнено и зафиксировано в Project Baseline 0.1
**Дата:** 2026-07-18
**Ограничение:** этот документ не перемещает и не удаляет существующие файлы.

## 1. Цель

Сделать документацию понятной в трёх режимах:

- быстро понять состояние проекта;
- принять или проверить инженерное решение;
- найти подробные правила и историю.

## 2. Предлагаемая целевая навигация

```text
docs/
├── executive/       # краткий управленческий контекст и ближайшие цели
├── core/            # vision, architecture, decisions и инженерные принципы
├── development/     # backlog, module/API specs, состояние и журнал разработки
├── standards/       # обязательные нормы процесса, кода, Git и документации
├── templates/       # канонические шаблоны
└── AI_TEAM/         # роли, рабочие протоколы и конституции
    └── constitutions/
```

Каталоги `workspace/` и `archive/` остаются вне `docs/`: это рабочие и
исторические материалы, а не нормативная документация.

## 3. Роли каталогов

| Каталог        | Содержит                                                        | Не содержит                                |
| -------------- | --------------------------------------------------------------- | ------------------------------------------ |
| `executive/`   | текущий контекст, состояние, активные решения, ближайшую веху   | подробные спецификации и историю изменений |
| `core/`        | продуктовое направление, архитектуру, ADR и инженерные принципы | ежедневные отчёты                          |
| `development/` | исполнимый бэклог, спецификации модулей/API, журналы            | глобальные стандарты                       |
| `standards/`   | обязательные правила                                            | временные договорённости конкретной задачи |
| `templates/`   | единственный канонический вариант шаблона каждого типа          | заполненные документы                      |
| `AI_TEAM/`     | роли, handoff, правила взаимодействия, конституции              | архитектурные решения продукта             |

## 4. Нормативные источники истины

В Project Baseline 0.1 закреплено следующее разграничение:

- Текущее состояние — один канонический документ в `executive/01_CURRENT_STATE.md`.
- Архитектурные решения — `docs/core/99_DECISIONS.md` или отдельные ADR-файлы.
- План верхнего уровня — `docs/core/01_MASTER_PLAN.md`.
- Исполнимые задачи — `docs/development/06_BACKLOG.md` и task-файлы.
- История релизных изменений — `docs/development/07_CHANGELOG.md`.
- Хронология рабочего процесса — `docs/development/15_PROJECT_JOURNAL.md` и
  `workspace/`.

## 5. Нормализованные пересечения

| Пересечение                                         | Предлагаемое направление                                             |
| --------------------------------------------------- | -------------------------------------------------------------------- |
| `04_PROGRESS.md` и `13_PROJECT_STATE.md`            | Progress показывает roadmap position; Project State — текущий факт   |
| `01_MASTER_PLAN.md` и `06_BACKLOG.md`               | Roadmap задаёт этапы; Backlog хранит active tasks                    |
| `05_AI_GUIDE.md` и `14_AI_CONTEXT.md`               | Guide задаёт чтение; Context — runtime ограничения                   |
| `11_WORKFLOW.md` и `12_PROJECT_OPERATING_SYSTEM.md` | Workflow нормативен; Operating System хранит workspace rules         |
| Два Task Template                                   | Канонический template находится в `docs/templates/`                  |
| Два Git-документа                                   | Git Standard задаёт правила; Delivery Policy описывает их применение |

## 6. Правило после baseline

Структура документации frozen. Массовый перенос, переименование или
переписывание разрешены только через ADR либо решение Product Owner с
измеримой пользой.
